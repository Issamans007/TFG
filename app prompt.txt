==================================================
OFFLINE SYNC STRATEGY
==================================================

Handle device internet disconnections gracefully:

Scenario: Device disconnects from internet
- Local trade queue buffers all signals received while offline
- Timestamp all local trades with device time
- Cache order status locally (pending execution)
- Monitor SL/TP locally without server confirmation
- Close positions locally if emergency triggered

Signal Handling While Offline:
- Store incoming signals in local SQLite queue
- Missed signals (arrived while offline) can be:
  * EXECUTED: If still valid (check expiry timestamp) when reconnected
  * SKIPPED: If signal expired during offline period
  * BATCHED: Multiple signals processed together on reconnect
- User config: auto-accept or manual-review missed signals on reconnect

Conflict Resolution on Reconnection:
- Compare local position state vs server state
- If local > server: position was closed server-side, revert locally
- If server > local: missed signals executed by server, sync order history
- If divergent: log conflict, notify user, prompt manual resolution
- Idempotent execution: signalId prevents duplicate trades
- Sync order history after each reconnect

Local Trade Queue Management:
- Priority queue: SL/TP orders > new entry signals
- Maximum queue size: prevent memory overflow
- Persistent storage: don't lose trades if app crashes
- Retry mechanism: exponential backoff to push queued trades
- Dead letter queue: store failed trades for manual review

Network Recovery:
- Detect reconnection via connectivity manager
- Auto-drain local queue when online
- Verify Binance API connectivity before execution
- Notify user of sync status (pending, syncing, synced)
